00068     // 4 bytes of closed caption, 2 from field 1 followed by 2 from field 2
00069     // When this is used with progressive clips either the two bytes of field 1
00070     // data or the two bytes of field 2 data should be 0xff,0xff or 0xfc,0xfc.
00071     // These special values which are normally invalid CC byte values because
00072     // the parity bit is incorrect are used to indicate "no data". For
00073     // extraction they mean no data was found. For insertion they mean do not
00074     // insert or insert invalid data as described below. (The two forms of
00075     // "no data" indication are needed to support proper insertion into

00077     //

00080     // 1. The four return bytes are initialized to 0xff 0xff 0xff 0xff
00081     // 2. The cc data triples (cc_type, cc_data_1, cc_data_2) are walked
00082     //    in order and each NTSC triple is processed. Walking stops when a
00083     //    DTVCC type triple is found.
00084     // 3. Each NTSC triple is processed as follows:
00085     //    a. Field 1 marked as invalid (cc_type is 0xf8) converts the first two
00086     //       returned bytes from 0xff to 0xfc. Do nothing if the first two

00091     //       bytes from 0xff to 0xfc. Do nothing if the second two return bytes

00097     // 1. Remove the NTSC cc data triples by walking the cc data triples
00098     //    converting NTSC triples to DTVCC channel packet data marked as
00099     //    invalid (cc_type is 0xfa). Stop walking when a DTVCC triple is
00100     //    found. Return failure if cannot convert one triple for progressive
00101     //    clips or two triples for interlace clips.
00102     // 2. Convert the first triple for progressive clips to NTSC cc data
00103     //    triples using the following rule.
00104     //    a. If the first two of the supplied four bytes are not 0xff,0xff or
00105     //       0xfc,0xfc then insert the two bytes as Field 1 data (0xfc)
